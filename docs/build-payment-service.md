# Build Plan: Payment Service

## Role in the System

The Payment Service is the most fault-prone component in the topology. It sits at the center of the pipeline: it consumes from `order-queue`, performs simulated payment processing (the most latency-sensitive step), and produces to `payment-queue` for Fulfillment. When it fails, it fails loudly and cascades in both directions — backing up `order-queue` behind it and starving Fulfillment and Notification downstream.

The Payment Service is also the origin of `payment-dlq`, where messages that fail processing after exhausting retries are deposited. DLQ depth is one of the primary anomaly signals the correlation engine watches.

---

## Happy Path Behavior

1. Polls `order-queue` for messages via `queue.receive('order-queue', { maxMessages: 5 })`
2. For each message, runs simulated payment authorization:
   - Validates message schema
   - Simulates a ~50–150ms processing delay (network call to payment provider in real life)
   - Produces a `PaymentMessage` result (approved/declined, transaction ID)
3. On success: enqueues result to `payment-queue` and deletes the source message from `order-queue`
4. On processing error (after retries): sends the original message to `payment-dlq`
5. Writes `service_metrics` and the queue writes its own `queue_metrics` rows every 2–3 seconds

Under normal conditions, the Payment Service is the throughput bottleneck — it processes slightly slower than the Order Service produces, but the queue absorbs the difference.

---

## Queue Interface

### Consumes

**Queue:** `order-queue`

Reads up to 5 messages per tick. On success, deletes the message (fake SQS `deleteMessage`). On failure after retries, forwards to DLQ.

### Produces

**Queue:** `payment-queue`

**Message shape:**

```
PaymentMessage {
  messageId:    string     // UUID
  orderId:      string     // from source OrderMessage
  customerId:   string     // from source OrderMessage
  totalAmount:  number     // from source OrderMessage
  status:       'approved' | 'declined'
  transactionId: string    // UUID, generated by payment processing
  processedAt:  number     // Unix ms
  traceId:      string     // propagated from OrderMessage
  processingMs: number     // time spent in payment processing step
}
```

**Queue:** `payment-dlq`

**Message shape:** original `OrderMessage` wrapped in a dead-letter envelope:

```
DLQMessage {
  originalMessage: OrderMessage
  failureReason:   string          // e.g. "schema_invalid" | "processing_error" | "timeout"
  attemptCount:    number
  firstAttemptAt:  number          // Unix ms
  lastAttemptAt:   number          // Unix ms
}
```

## Retry Logic Simulation

The Payment Service implements a simple retry policy before sending a message to the DLQ:

- **Max attempts:** 3
- **Backoff:** fixed 500ms between retries (not exponential — this is a simulation; exponential backoff would slow the demo significantly)
- **Retry condition:** any processing error except schema validation failures (those go directly to DLQ on first attempt)
- **Visibility timeout simulation:** a message that is being retried is marked as "in-flight" and not re-polled by another worker tick

The retry counter is tracked in an in-memory `Map<messageId, attemptCount>`. This is acceptable for a simulation — in production, the attempt count would be tracked in the message attributes or a separate store.

---

## Timeout Handling

The downstream timeout fault injects `processingDelayMs = 8000` into the Payment Service's fault context. The service simulates this by calling `time.Sleep(time.Duration(processingDelayMs) * time.Millisecond)` inside the processing goroutine.

Because each processing tick holds an "in-flight" lock on the messages it picked up, the effect is:
- The Payment Service appears to be working (it has messages in flight)
- It never completes them
- New messages are not picked up because the in-flight limit is reached
- `order-queue` depth climbs
- The messages time out and are retried; after 3 attempts they go to DLQ
- Downstream services (Fulfillment, Notification) see no new messages — starvation begins

The timeout duration (8000ms) is deliberately longer than the retry backoff window, which means most messages will exhaust all retries before the timeout resolves.

**Observable metrics during timeout fault:**

| Metric | Expected value |
|---|---|
| `p99_latency` | Climbs toward 8000ms |
| `throughput` | Drops to near zero (processing bottlenecked) |
| `error_rate` | Rises as retries exhaust and messages DLQ |
| `payment-dlq depth` | Climbs steadily |
| `order-queue depth` | Climbs as producer continues at normal rate |

---

## DLQ Behavior

The `payment-dlq` queue is a standard queue with no consumer (in production, a DLQ re-drive operation would be run manually). In this simulation:

- Messages written to `payment-dlq` stay there until the simulator is reset
- The queue writes `queue_metrics` rows with `dlq = 1`
- DLQ depth above 0 is an immediate anomaly signal for the correlation engine — there is no baseline to establish, any positive depth is flagged
- The dashboard shows DLQ depth as a red badge on the Payment Service card

## Fault Behavior

### Poison Pill

- `order-queue` delivers messages with `_meta.poisoned = true`
- Payment Service's schema validator detects a poisoned message flag and generates a processing error
- Because `poisoned = true` is treated as a schema error, messages go directly to DLQ (no retry)
- `error_rate` climbs toward 1.0 very quickly
- `payment-dlq` depth climbs proportionally to throughput
- `payment-queue` stops receiving messages — Fulfillment begins to starve

### Downstream Timeout

- `faultContext.processingDelayMs = 8000`
- See "Timeout Handling" section above

### Traffic Spike

- `order-queue` depth climbs 10x faster than normal
- Payment Service cannot scale (it's a fixed-rate worker) — it processes at its normal rate
- Queue depth backs up significantly; queue latency rises
- Payment Service metrics themselves remain normal (no errors, normal latency per message) — the anomaly is in the queue, not the service

### Cascading Failure

- `faultContext.shouldCrash = true`
- The Payment Service's `tick()` method throws an unhandled error
- The simulator catches this at the worker level and marks the service as `status: 'crashed'`
- No further messages are consumed from `order-queue`
- No messages are produced to `payment-queue`
- `throughput` drops to zero; the metric writes continue (the metrics loop is separate from the processing loop)
- Downstream starvation begins immediately

**Recovery:** the fault engine's `deactivate('cascading-failure')` call sets `shouldCrash = false` and calls the service's `restart()` method, which re-initializes the processing loop.

### Random Intermittent Errors

- `faultContext.errorProbability = 0.15`
- On each message, a random draw determines if processing "fails" (simulates transient errors)
- Failed messages are retried; most succeed on retry (retry logic rolls the dice again)
- Net effect: slightly elevated `error_rate` (~0.05–0.10 after retry success rate), slightly elevated `p99_latency` due to retry wait
- Small number of messages exhaust retries and go to DLQ; DLQ accumulates slowly

---

## Implementation Notes

- The Payment Service runs a goroutine driven by a `time.NewTicker` at ~100ms ticks, polling up to 5 messages per tick (effective throughput: up to 50 messages/s, well above normal load)
- The processing delay simulation calls `time.Sleep()` inside the per-message goroutine so it doesn't block other service tickers; all services run in independent goroutines
- The in-flight message map (`map[string]int` keyed by messageId) is bounded to prevent unbounded growth during sustained fault scenarios
- Fault context is read once per tick (not per message) to avoid inconsistent behavior mid-batch

## Acceptance Criteria

| Scenario | Observable outcome |
|---|---|
| Happy path 60s | `throughput` ≥ 4 msg/s; `error_rate` ≤ 0.01; `payment-dlq` depth = 0; `payment-queue` depth ≤ 3 |
| Poison pill activated | `error_rate` → ≥ 0.9 within 3 metric cycles; `payment-dlq` depth > 0 within 10s; Fulfillment throughput → 0 within 15s |
| Downstream timeout activated | `p99_latency` → ≥ 5000ms within 10s; `order-queue` depth rising; `payment-dlq` depth rising after retry exhaustion |
| Cascading failure activated | `throughput` → 0 within one metric cycle; `service_metrics.active_faults` includes `'cascading-failure'`; downstream starvation confirmed in Fulfillment metrics |
| DLQ behavior | Any message that fails 3 processing attempts appears exactly once in `payment-dlq`; never reprocessed |
| Fault deactivated | Within 10s of deactivation, metrics begin recovering; `status` transitions from anomaly back to baseline |
