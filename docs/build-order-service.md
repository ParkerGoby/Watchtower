# Build Plan: Order Service

## Role in the System

The Order Service sits immediately downstream of the API Gateway. It is the entry point into the queue-based processing pipeline. It receives order requests, validates them (lightly), serializes them to a message payload, and enqueues them onto `order-queue` for the Payment Service to consume.

In the happy path, the Order Service is a throughput translator: it converts HTTP-style request objects (generated by the simulated API Gateway) into queue messages at a controlled rate.

---

## Happy Path Behavior

1. API Gateway generates a synthetic order request (item ID, quantity, customer ID, price) at a configurable rate (default: ~5 req/s)
2. Order Service receives the request, runs shallow validation (non-null fields, price > 0)
3. Serializes the order into a `OrderMessage` payload
4. Calls `queue.send('order-queue', payload)` — the fake SQS `sendMessage` implementation
5. Increments its internal throughput counter
6. Writes a `service_metrics` row to SQLite every 2–3 seconds

Under normal conditions, `order-queue` depth stays near zero because the Payment Service consumes messages roughly as fast as they arrive.

---

## Queue Interface

### Produces

**Queue:** `order-queue`

**Message shape:**

```
OrderMessage {
  messageId:  string       // UUID
  orderId:    string       // UUID
  customerId: string       // UUID
  items: Array<{
    itemId:    string
    quantity:  number
    unitPrice: number
  }>
  totalAmount: number
  createdAt:   number      // Unix ms
  traceId:     string      // propagated through the full pipeline for correlation
  _meta: {
    poisoned:  boolean     // set to true by the fault engine for poison pill scenario
    emittedAt: number      // Unix ms, for latency measurement downstream
  }
}
```

The `_meta.poisoned` flag is how the poison pill fault is implemented: the Order Service sets it to `true` on affected messages, and the Payment Service reads it and generates a processing error.

### Consumes

None. The Order Service does not read from any queue.

---

## Metrics Emitted

The Order Service writes one row to `service_metrics` every 2–3 seconds (jittered to avoid synchronized writes from all services).

| Column | How it's calculated |
|---|---|
| `service` | `'order'` |
| `ts` | `Date.now()` |
| `throughput` | Messages enqueued since last write ÷ elapsed seconds |
| `error_rate` | Validation errors since last write ÷ total requests since last write |
| `p50_latency` | Median time from request receipt to `queue.send()` return, in ms |
| `p99_latency` | 99th-percentile of same |
| `active_faults` | Comma-separated list from `faultEngine.getActiveFaults()` |

The Order Service does **not** directly write `queue_metrics` — the queue object itself writes those rows.

---

## Fault Behavior

### Poison Pill

- The fault engine sets `faultContext.poisonedMessageRate = 1.0` for the order service
- On each message serialization, the Order Service calls `faultEngine.getFaultContext('order')`
- If `poisonedMessageRate > 0`, a random draw determines whether the message gets `_meta.poisoned = true`
- At rate 1.0, every message is poisoned
- The Order Service's own metrics will look normal — error rate stays low, latency stays low, throughput stays normal
- The impact manifests downstream in the Payment Service

### Downstream Timeout (indirect)

- The Payment Service is hanging; it stops consuming from `order-queue`
- The Order Service keeps enqueueing at normal rate
- `order-queue` depth begins to climb
- Order Service `throughput` metric stays high (it is still producing), but this combined with rising queue depth is the anomaly signal the correlation engine picks up

### Traffic Spike

- The API Gateway fault context sets `throughputMultiplier = 10.0`
- API Gateway generates 10x the normal request rate
- Order Service receives and processes all of them
- `throughput` in `service_metrics` spikes dramatically
- If Order Service cannot keep up, its own `p99_latency` rises as its internal processing loop falls behind
- `order-queue` depth rises because Payment Service cannot absorb the burst

### Cascading Failure (indirect)

- Payment Service crashes; `order-queue` depth rises as with the timeout scenario
- Order Service metrics otherwise remain normal — it is a healthy producer in a system where the consumer has died

### Random Intermittent Errors

- `faultContext.errorProbability = 0.15`
- Order Service simulates a ~15% validation failure rate by randomly discarding messages and recording them as errors
- `error_rate` in `service_metrics` fluctuates around 0.15
- This is below the anomaly threshold on its own; only when correlated with similar signals from other services does the monitor flag it

---

## Implementation Notes

- The service runs as a Go struct with a goroutine driven by a `time.NewTicker` at ~200ms intervals (simulating 5 req/s). Lifecycle is controlled via a `context.Context` — cancelling the context stops the goroutine cleanly.
- Processing rate is configurable at startup; the default is tuned so that `order-queue` stays near empty under normal load
- Internal metrics are accumulated in memory between SQLite writes to avoid a DB write on every message
- The `traceId` field is a UUID generated at the API Gateway and passed through the entire message chain — this is what allows the correlation engine to link a fulfillment failure back to a specific order

---

## Acceptance Criteria

| Scenario | Observable outcome |
|---|---|
| Happy path running for 60s | `service_metrics` has ~30 rows for `service='order'`; `error_rate` ≤ 0.01; `queue_metrics` for `order-queue` shows depth ≤ 5 |
| Poison pill activated | Order Service metrics look normal; Payment Service `error_rate` rises to > 0.8 within 10s |
| Traffic spike activated | Order Service `throughput` jumps 10x within one metric write cycle; `order-queue` depth begins rising |
| Downstream timeout activated | `order-queue` depth rises steadily; Order Service `throughput` remains stable |
| Service stopped cleanly | `stop()` called; no further rows written; in-flight messages drained or discarded gracefully |
